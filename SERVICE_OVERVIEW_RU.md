# Marketing Service: текущее состояние сервиса

Этот документ описывает, что из себя представляет сервис **на текущем этапе разработки**: его архитектуру, бизнес-логику, взаимодействие с LLM/«агентами» и текущие ограничения.

## 1. Что это за сервис

Сервис предназначен для editorial/publishing workflow:

1. Создаётся **проект**.
2. Внутри проекта создаются **исходные статьи**.
3. Для каждой статьи создаются **адаптации под каналы**:
   - Telegram
   - X
   - Discord
4. Для адаптаций при необходимости создаются **переводы** на другие языки.
5. После генерации и ручного утверждения контента формируются:
   - либо **publication plans** (план публикации),
   - либо сразу **scheduled publications** (запланированные публикации с автопостингом).
6. Сервис сам публикует готовые посты в целевые каналы в назначенное время.

Дополнительно на уровне проекта есть сущность **draft markers**: это предварительные маркеры/идеи публикаций, которые можно расставлять по календарю до создания полноценной статьи.

## 2. Архитектура

### Общая структура монорепы

Монорепа построена на `pnpm workspace` + `turbo`.

Основные директории:

- `apps/api` — HTTP API и встроенный test UI.
- `apps/worker` — заготовка под отдельный worker-процесс.
- `packages/shared` — базовые DDD-примитивы.
- `packages/database` — Drizzle schema и миграции.
- `packages/project-management` — домен проектов и draft markers.
- `packages/editorial` — домен статей, адаптаций и переводов.
- `packages/publishing` — домен публикаций, расписания и автопубликации.
- `packages/infrastructure` — реализации портов: БД, LLM, Telegram, Discord, X.

### Текущая прикладная архитектура

Сервис по сути организован как **modular monolith** с элементами:

- DDD
- CQRS (`CommandBus`, `QueryBus`, handlers)
- Ports & Adapters
- NestJS modules

То есть:

- **доменные пакеты** описывают сущности, use case-ы и интерфейсы-репозитории/порты;
- **infrastructure** подставляет реальные реализации;
- **apps/api** даёт HTTP-вход;
- **apps/worker** пока почти не задействован.

### Технический стек

- `NestJS`
- `Fastify`
- `PostgreSQL`
- `Redis` (скорее как база для очередей/фоновой работы, но полноценный queue-flow ещё не развёрнут)
- `Drizzle ORM`
- `TypeScript`
- `SWC`
- `pnpm`, `turbo`

## 3. Основные bounded contexts

### 3.1 Project Management

Отвечает за:

- проекты;
- draft markers;
- размещение marker-ов в календаре.

Ключевые сущности:

- `Project`
- `ProjectMarker`
- `ProjectMarkerPlacement`

Смысл marker-ов:

- marker — это ещё не статья и не публикация;
- это предварительная идея публикации;
- marker можно поставить в несколько дат/каналов/языков;
- он помогает сначала разметить календарь, а потом создавать полноценный editorial flow.

### 3.2 Editorial

Отвечает за:

- исходную статью;
- channel adaptations;
- adaptation versions;
- translations.

Ключевые сущности:

- `Article`
- `ChannelAdaptation`
- `AdaptationVersion`
- `Translation`

Базовая модель:

- `Article` хранит оригинальный текст, язык, статус и snapshot release-плана.
- Для статьи можно создать несколько `ChannelAdaptation` — по одному на каждый канал.
- Для адаптации генерируются версии (`AdaptationVersion`).
- Для адаптации можно делать переводы (`Translation`) на другие языки.

### 3.3 Publishing

Отвечает за:

- планы публикаций;
- реальные scheduled publications;
- автопубликацию по времени;
- отмену и перенос публикаций.

Ключевые сущности:

- `PublicationPlan`
- `Publication`

Разница:

- `PublicationPlan` — это просто план/намерение.
- `Publication` — это уже сущность, которая реально должна быть опубликована автоматически.

## 4. HTTP-слой

### Основные контроллеры

- `ProjectController` (`/projects`)
- `ArticleController` (`/articles`)
- `PublicationController` (`/publishing`)
- `TestUiController` (`/test-ui`)

### Что важно понимать

`/test-ui` — это не отдельный frontend-проект.  
Это server-rendered HTML, встроенный прямо в `apps/api/src/editorial/test-ui.controller.ts`.

То есть текущий UI:

- рендерится на сервере;
- использует inline CSS;
- использует vanilla JS;
- нужен как рабочий операционный интерфейс/прототип, а не как отдельный production-grade SPA.

## 5. Текущая бизнес-логика

### 5.1 Жизненный цикл статьи

На текущем этапе flow выглядит так:

1. Пользователь создаёт `Article` из исходного текста.
2. Для статьи создаются адаптации под конкретные каналы.
3. Для адаптации вызывается генерация текста через LLM.
4. Генерация создаёт новую `AdaptationVersion`, а сама адаптация получает статус `generated`.
5. Пользователь может:
   - отредактировать текст вручную;
   - выбрать другую версию;
   - дать instruction на revision выделенного фрагмента;
   - approve адаптацию.
6. После approve остаётся только утверждённая версия; лишние версии удаляются.
7. Для адаптации можно создать переводы на другие языки.
8. Переводы тоже проходят через генерацию, ручную правку и approve.

### 5.2 Статусы editorial-узлов

Для `ChannelAdaptation` и `Translation` используется статусная модель:

- `pending`
- `generated`
- `edited`
- `approved`
- `outdated`

Смысл:

- `pending` — ничего ещё нет;
- `generated` — текст сгенерирован LLM;
- `edited` — текст менял человек;
- `approved` — текст утверждён и может использоваться для публикации;
- `outdated` — версия устарела из-за последующих изменений.

### 5.3 Публикация

Для автопубликации сервис требует, чтобы:

- нужная адаптация была `approved`, если публикация идёт на исходном языке;
- нужный перевод был `approved`, если публикация идёт на другом языке.

При публикации по расписанию сервис:

1. Ищет `Publication` со статусом `scheduled` или `failed`, которые уже пора отправлять.
2. Меняет статус на `publishing`.
3. Достаёт финальный publishable text:
   - либо approved adaptation,
   - либо approved translation.
4. Отправляет текст в нужный адаптер:
   - Telegram
   - Discord
   - X
5. Если всё успешно:
   - статус становится `published`.
6. Если ошибка:
   - статус становится `failed`,
   - сохраняется `errorMessage`.

## 6. Взаимодействие с агентами / LLM

### Что здесь считается «агентом»

На текущем этапе в системе нет сложной многоагентной оркестрации внутри backend-а.  
Под «агентной» логикой здесь фактически понимается **LLM-слой**, который делает:

- генерацию адаптаций;
- revision выделенного фрагмента адаптации;
- перевод адаптаций.

### Как это реализовано

В домене editorial есть порты:

- `AdaptationGeneratorPort`
- `TranslationGeneratorPort`

В `infrastructure` эти порты сейчас реализованы классом:

- `DeepSeekAdaptationGenerator`

Он работает как единый LLM-adapter и умеет:

- `generate(...)`
- `reviseSelection(...)`
- `translate(...)`

### Как он используется

Flow примерно такой:

1. Handler собирает доменный контекст:
   - оригинальный текст;
   - язык;
   - channel id;
   - display name;
   - prompt instructions.
2. Этот контекст передаётся в LLM adapter.
3. LLM возвращает готовый текст.
4. Handler сохраняет результат в доменные сущности.

### Особенности промптинга

Промпты отличаются по каналам:

- **Telegram**: короткий, читаемый, 3 предложения.
- **X**: 1 предложение, до 15 слов, очень сжато.
- **Discord**: максимально просто и понятно, до 2 предложений.

Если для канала заданы `promptInstructions`, они имеют приоритет и вшиваются в system prompt.

### Важный вывод про «агентность»

Сейчас система — это не «автономный агент, который сам принимает бизнес-решения».  
Это скорее **human-in-the-loop editorial pipeline**, где LLM играет роль умного генератора текста, а финальное решение остаётся за человеком.

Иными словами:

- AI предлагает текст;
- человек редактирует, утверждает и планирует публикацию;
- система публикует только утверждённое.

## 7. Интеграции

### Telegram

Есть отдельный адаптер `TelegramBotApiPublisher`.

Сервис публикует сообщения через bot token / chat id, зависящие от языка.

### Discord

Есть `DiscordWebhookPublisher`.

Публикация идёт через webhook URL.

### X

Есть `XApiPublisher`.

Текущая интеграция работает через:

- `POST /2/tweets`
- OAuth 1.0a user context

Сейчас поддерживается только **текстовая публикация** в X.  
Медиа-вложения пока не реализованы.

## 10. Что пока недоделано или сделано частично

### Worker

`apps/worker` существует, но фактически пока это заготовка.  
Основная реальная фоновая логика сейчас живёт внутри `api` процесса:

- scheduler публикаций запускается в `PublishingModule` внутри API.

То есть отдельного production-grade разделения на:

- API
- background workers
- queues/processors

пока ещё нет.

### Очереди и health-checks

В `apps/api` и `apps/worker` есть следы будущей инфраструктуры:

- queue modules
- health modules
- processors

Но значительная часть этого пока закомментирована или не подключена.

### UI

UI существует в виде встроенного `test-ui`, а не отдельного frontend-приложения.  
Он рабочий, но архитектурно это временный/операционный интерфейс, а не окончательный production frontend.

### Marker → Article automation

Идея уже есть: создавать статью на основе marker-а и его расставленных placement-ов.  
Но на текущем этапе это ещё не законченный автоматизированный flow.

## 11. Ключевая мысль о сервисе в одном абзаце

Сейчас это **монолитный editorial orchestration service** для ручно-управляемого контентного pipeline: пользователь создаёт статью, генерирует канал-специфичные адаптации и переводы через LLM, вручную утверждает их и затем планирует или автоматически публикует контент в Telegram, Discord и X. Архитектурно сервис уже разложен на доменные модули и порты/адаптеры, но всё ещё находится на стадии pragmatic working prototype: UI встроен в API, worker-контур заготовлен лишь частично, а основная ценность сейчас сосредоточена в связке `project -> article -> adaptation -> translation -> publication`.

## 12. Если другой модели нужен совсем короткий summary

Можно описать сервис так:

> Это NestJS-based modular monolith для управления editorial workflow. Он хранит проекты, статьи, адаптации под каналы, переводы, планы публикаций и реальные scheduled publications. Генерация адаптаций и переводов делается через DeepSeek LLM adapter. После ручного approve сервис автоматически публикует контент в Telegram, Discord и X. UI пока встроен прямо в API как server-rendered test dashboard. Draft markers уже существуют как отдельная проектная сущность для предварительного планирования контента в календаре.
